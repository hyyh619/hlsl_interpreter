0 · 项目概述

本项目是一个纯 Python 实现的 Direct3D 11 图形管线软件仿真器，核心特点是直接解释执行 HLSL 着色器源码——不编译、不调用 eval，而是自己做词法/语法分析并逐语句解释。它把一帧真机捕获的 GPU 数据喂进这条仿真管线，跑完整的固定功能 + 可编程阶段，再把顶点着色器输出与真机捕获的 golden 参考数据逐分量对比，以此验证解释器的正确性。

蓝实线 = 管线数据流；绿虚线 = VS 输出与 golden 的验证对比

输入从哪来

输入是 RenderDoc / 3Dmigoto 抓帧 dump，打包成 Cases/*.zip。每个 zip 解压后是一组松散文件：

文件作用
VS_shader.hlsl / PS_shader.hlsl着色器源码（main() 用带语义的参数声明输入输出）
*_input_output_signature.csv参数 ↔ 寄存器槽位映射
ia_vertex_data.csv + 二进制 VB/IB输入装配阶段的顶点 / 索引数据
*_constant_buffers.csv + constant_*.bin常量缓冲区（文本布局 + 二进制精确值）
pipeline_state.csv光栅化 / 混合 / 深度模板状态 + 图元拓扑
output_merger.csv + pre_draw_ds_res_*.raw输出合并阶段 RTV/DSV 布局与 draw 前深度模板内容
MeshOut_*_mesh.csvgolden VS 输出（验证基准）
.img / BMP 纹理Texture2D.Sample(...) 的后备数据

怎么运行 · 模块构成

唯一入口是一个 JSON 配置：python render.py ./Cases/Default.json。render.py 是编排器，在 zip 工作流（当前主路径）与 legacy struct 工作流（旧配置兼容）间分支。零第三方依赖——只用标准库，外加可选的 tkinter（网格查看器）。

模块职责
hlsl_interpreter.py（~3300 行，核心）解析 HLSL（cbuffer/struct/函数/纹理采样器绑定），遍历语句、求值表达式
hlsl_syntax_tree.py手写递归下降表达式解析器，产出 SyntaxTreeNode 树；运算符优先级表在此
rasterizer.py按图元拓扑光栅化；边函数 + 透视正确重心插值；视口变换与剔除
output_merger.py深度 / 模板测试，支持 early-Z / late-Z
texture.py纹理解析、mipmap、采样与寻址模式
pixel.py / d3d.py / mesh_view.py像素交换记录 / D3D 枚举常量 / 可选网格查看器
正确性如何保证：项目没有传统单元测试，靠两条防线。① verify-by-log：跑管线 → grep output.log 里的 Error: 行（VS-vs-golden 分量级不符）→ 迭代到无错；[STMT] / [BINARY OP] 轨迹是调试主面。② 数据驱动回归套件：run_regression.py 逐个 headless 跑 regression_test_zip_files.csv 列出的 capture，三条判据（干净退出、无 Error:、Total PASSED rows: X/X）全满足才算过，目前覆盖 123 个真机 case，是每次改动的提交闸门。本手册后续各章即围绕"如何与 AI 协作、在这样一个无单测、靠真机 golden 验证的解释器项目上持续修 bug"展开。

1 · 如何给 AI 提供提示词

提示词不是"一句话指令"，而是一个分层的上下文系统。本项目里有效的提示词由四层组成：

提示词四层：越靠上越"常驻"，越靠下越"本轮"
要点：把"踩过的坑"和"必须遵守的纪律"写进常驻说明（CLAUDE.md），AI 才不会反复犯错。C1 修复时 AI 直接知道要看 golden 对比日志、改完跑回归——都来自 CLAUDE.md。给提示词的黄金法则：给可机判的成功标准（日志无 Error: 且 Total PASSED rows: X/X），指明数据/代码在哪，写明不要做什么。

2 · 分析需求、生成开发计划

面对"挨个修复 issue"这种开放任务，AI 先 侦察(triage) → 聚类 → 排序 → 立计划，而不是盲目动手。

侦察：把"未知"变"已知"

Dump 目录 355 个 zip，其中 witcher3 有 160 个、仅 31 个在回归、129 个从未验证。AI 写并行 triage 脚本，把每个 zip 跑成 headless 管线并分类 PASS/FAIL/CRASH/NOCMP/TIMEOUT。

聚类：129 个失败 → 少数根因类

问题类代表 case处置
sv_position/TexCoord 读成 023341/23141/23251/23183/23195一个 bug(C1) 一次修 5 个
大气输出错20899C2 已修
(int) 强转 / 顶点纹理取样16834/16215C3 修 + 受阻
raw buffer 未 dump22201受阻：capture 限制
慢绘制~26 + Tankperf，非 bug
关键洞察 — 先找杠杆点：聚类让"129 个失败"坍缩成"少数根因"。修一个根因(C1)就让 5 个 case 同时变绿。AI 用 TodoWrite 把计划落成可勾选清单，并按 ROI（确定性高、风险低、影响面大） 排序：先 C1，再 C2，再谨慎做 C3，受阻类诚实暂缓。

3 · 如何按计划执行

每个 case 走同一个闭环：

回归不绿不提交；提交后把代表 case 加进回归集，循环下一个

以 C1 为例：沿数据流回溯

从错误输出值逆推到那行 break：

Error: Row 0 TexCoord[0]: output=0.000000 golden=0.129650   ← 输出恒为 0
   ↑ o1.x = dot(v1, texgen[0])              （读 VS_shader.hlsl）
   ↑ texgen 是 float4 texgen[2] : packoffset(c2)
   ↑ 组合 CSV 每个数组名只有一个代表值 → texgen 加载成 [None,None]
   ↑ override_cbuffers_from_binary:
        for field in cb_def.fields:
            if field.array_size > 0:
                field.data = decoded[:m]
                break          # ← 只填第一个数组就退出！texgen 仍是 [None,None]

修复：解析 packoffset(cN)、新增 _field_register_offsets()、重载循环改为给每个数组按各自寄存器窗口 decoded[reg_off:reg_off+size] 填充。验证：23341 由 0→30/30，另 4 个同类全过；回归 44/44；提交 1716ee5。

4 · 输入 / 输出数据流

AI 的输入是累积上下文；输出是文字或工具调用；工具结果回流形成闭环

项目自身的数据流（被解释的对象）

zip(capture) → 解压 → CSV / HLSL / BMP / bin
   ├─ ia_vertex_data + 二进制 VB ─┐
   ├─ cbuffers(CSV) + constant_*.bin ─┤   ← C1 在这段；C2 在 .bin 解码
   └─ VS_shader.hlsl ───────────────┤
                                     ▼
        VertexShader 解释执行 → VS 输出 → 与 golden(*_vs_mesh.csv) 对比
                                     ▼
        Rasterizer → Depth/Stencil → PixelShader → OutputMerger

5 · 什么样的数据流输出能帮助调试

结论：能把"中间状态"暴露成可 grep 的文本，最有价值。本项目刻意把解释器做成"会自述"的。

C3 的 smoking gun — 逐语句轨迹

[STMT] r0.xy = (int2)v1.zw => r0.xy = [1073741824, 1073741824]   ← 把 2.0 当位型！应为 [2,2]
[METHOD] t1.Load((1073741824,0,0)) = ['0','0','0','0']             ← 越界读 0
[BINARY OP] left=['312.5','500.0'] right=['0','0'] op=/ => ['inf','inf']   ← 0/0
... → sv_position = nan

正是这条轨迹让 C3 一眼可见：(int2)v1.zw 把 2.0 变成它的位型 1073741824 而不是 2。只看最终 nan 是无从下手的。

分量级报错 + 可机判汇总

Error: Row 0 TexCoord2[0]: output=2.466900 golden=0.679580 diff=1.787320
Total PASSED rows: 0/2160   →（修复后）2160/2160
C2 全靠轨迹：event20899 一度被误判"精度不可解"。重新打开逐语句 + 逐分支轨迹，把 o0/o1 回溯到 cb12[271].z ? r2.x : 1 分支选错，再发现 cb12[271].z 字节是 0x00000001（反规格化数 1.4e-45）。没有逐步轨迹，这个根因不可能被发现。

设计准则：让被测系统打印"每一步的输入、运算、输出"，且保证可被 grep Error: / grep '[STMT]' 提取。可观测性 = 可调试性。

6 · 回归测试如何建立与应用

数据驱动
regression_test_zip_files.csv 一行一个必过的 capture zip。
headless 跑
run_regression.py 逐个跑 render.py，写 per-case 日志，打印 PASS/FAIL，失败则退出码非零。
三条通过判据
① 干净退出 ② 无 Error: 行 ③ Total PASSED rows: X/X
改动回归结果动作
C1 多数组 cbuffer44/44提交 + 加入 event23341
C2 FTZ45/45 → 46/46（含新加 20899）提交 + 加入 event20899
C3 ftoi（改变既有语义，高风险）46/46回归绿才敢提交
每修好一类就把代表 case 加进回归 → 这两类永不回退。回归是"提交闸门"，尤其保护 C3 这种改变既有语义的高风险改动。（回归 CSV 与 zip 是 gitignore 的本地基础设施。）

7 · TDD 在本项目的应用

本项目是 TDD 的变体 —— "golden 即测试"：测试（GPU 真机 *_vs_mesh.csv）先于实现存在且必然正确。

先让失败可复现可量化（红），写最小修复（绿），加回归防回退

与单元 TDD 的差别：测试用例不是手写断言，而是真机 golden 数据；框架是 verify-by-log + 回归脚本。但红绿节奏、"测试驱动最小实现"、"测试防回归"三大要义完全一致。C1/C2 都是先确认"红"（读成 0 / 0 通过），再写最小修复转"绿"。

8 · 如何纠正 AI 的错误

① 自我纠错
step119 把 C2 判为"精度不可解"。step120 回到逐语句轨迹深挖，发现真因是反规格化数分支——从"不可解"变"一行修复"。
② 客观裁判
C3 改语义前先证伪担忧：翻出 event1031/1399 的 v1 是 uint 声明（不受影响），再改；回归 46/46 复核。
③ 护栏固化
把历史坑写进 CLAUDE.md（golden 尾部 float3 错位、一元/二元 +/-），AI 不再重犯。
④ 用户直接纠偏：用户在提示词里预标"哪些是 capture 限制、无解"，提前纠正 AI"什么都想修"的倾向，避免在死路上空转。
纠错的核心：回到数据流、要更细的证据，不要停在第一个看似合理的结论。

9 · 记忆系统如何避免重复开发

记忆系统 = memory/ 下"一事一文件" + 一个会话开始即载入的索引 MEMORY.md。

记忆内容避免的重复劳动
golden-vs-mesh-sv-position-firstgolden CSV 把 SV_Position 列排最前，按位置而非表头映射C2 分析时无需重推列序，直接确认 o6 对得上
per-vertex-binary-vb-decode-r10a2NORMAL/TANGENT 在二进制 VB 是 R10G10B10A2不必重新逆向顶点格式
raw-img-texture-loading纹理按 DXGI 从 .img 读、BC7 回退 BMP直接复用
witcher3-array-cbuffers-…数组 cbuffer 与实例化输入的坑C1 的背景知识
当一个"非显而易见、未来还会用到"的事实被发现，就写成一条记忆（带"为什么/怎么用"），并在索引留一行指针。同一个坑只踩一次。纪律：不记录代码/git 已记录的东西；先查重再写；召回的记忆引用到具体文件/函数时先核实仍存在。

10 · 两个 Agent 开发周期的对比

本项目历经两个 agent 周期（约 step1→step123，2026-06-05 起）。以下基于 Sessions/ 日志与提交历史的可见证据。

维度前期 · minimax-m2.7后期 · opus
目标从无到有搭骨架真机数据攻坚长尾

两周期互补：前期的小步与可观测性铺垫（那些 [STMT]/[BINARY OP] 打印），正是后期能"沿数据流快速回溯根因"的前提。没有前期的轨迹输出，后期的 C2/C3 根因不可能被一眼看穿。
