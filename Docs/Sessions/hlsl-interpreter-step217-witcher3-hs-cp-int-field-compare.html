Step 217 — witcher3 曲面细分 cp[4]：修复 CP-流比较对整型 VS 输出字段的误判

Date: 2026-07-21

Prompts（本步任务）

修复 witcher3 曲面细分 event21346：按 DS-out 布局比较 HS 控制点流，失败集中在 cp[4]（=CP 记录第 4
分量）：ours 1.0 vs golden 0.0，169/1024 行。

结论先行

根因是 CP-流比较（render.py）对整型 VS 输出字段的处理错误，不是解释器/着色器 bug。 cp[4] =
VS 输出 o1.x（int4 o1 : TEXCOORD1，地形 LOD 瓦片索引）。GPU 写的是整数，golden dump 存的是其
原始 32 位；比较侧按 DS-out 布局把该字节当 float 解码（int 1 → asfloat=1.4e-45，打印成
"0.000000"），再与我方整数值 1.0 比较 → 误报不匹配。我方 o1.x 经原始 DXBC 反汇编逐条验证是正确的。
修复比较：整型字段按位重解释对齐。event21346 855→1024/1024，全回归 191/191 全绿（零回归）。

执行 · 逐层定位

1. cp[4] = o1.x（整型瓦片索引）

VS 输出 dcl：o0.xyzw(4)、o1.x(1)、o2.xyzw(4)、o3.xyzw(4)。CP 记录扁平化后 cp[4] = slot1 =
o1.x（TEXCOORD1，int4）。o1.x = r1.x：地形瓦片搜索循环的结果（命中瓦片索引，或未命中 = -1）。

2. 我方 o1.x 正确（DXBC 逐条验证）

失败点（如 row615 pt=(131.25,−318.75)），且 o0（点坐标）与 golden 逐位相等（dx=dy=0）。查原始
VS_shader_disasm.txt：resinfo→3 瓦片、ld_indexable t17 取瓦片界、lt（严格 <）逐分量判定——与我方
执行完全一致。点 (131.25,−318.75) 不在 tile0 [257.8..757.8]，在 tile1 → o1.x=1。我方 =1 正确。

3. golden 其实也是 1（被误当 float）

直接解码 golden *_ds_mesh.bin row615 的 o1.x 字节：as_int=1、as_float=1.401298e-45。即 golden
存的整数就是 1（与我方一致！），但比较侧把它读成 float 1.4e-45（≈0.0），与我方 1.0 比 → 报
"golden=0.000000" 假失败。row0 等 golden=-1(int)=asfloat→NaN，靠既有 NaN 短路"碰巧通过"。

4. 修复：整型字段按位重解释比较

render.py CP-流比较：float 容差不过时，增补——若 fa 近整数且 int(round(fa)) == asint(golden_bits)
（struct.unpack('<i', pack('<f', fb))）则判过。即"GPU 写整数、golden 存原始位、我方持整数值"三者按位对齐。
该增补只在 float 比较失败后追加匹配条件，不会把已通过的用例变失败（对 190 既有用例是 no-op）；且
仅当 golden 位型恰为小整数、我方值恰等于该整数时才匹配，对真实 float 误差不会误判。

结果

  • event21346：855/1024 → 1024/1024（0 Error）；拷入 Cases/ + 入回归（190→191），删除 Dump 副本。
  • 全回归 191/191 全绿（含共用该比较的 sekiro4 曲面细分用例，零回归）。Dump 55→54。
  • 其余 witcher3（21719/21895/22260/23141、fighting_18028）是不同失败模式（TEXCOORD4 数值发散 ~0.1，属更深的 HS/DS 细分计算分歧，非本 cp[4] 整型比较 bug）；22229 为精度近过（10/12，step212 记录）。

关键结论

  • 该 cp[4] 失败是比较工具对整型 VS 输出字段的位/值处理错误，非解释器 bug——我方输出经原始 DXBC 反汇编 验证正确，golden 存的整数也与我方一致，只是被 DS-out 布局当 float 解码。
  • 修复通用且安全：整型字段按位重解释对齐；只增匹配、不破坏既有绿线。
  • witcher3 其余为独立的深层细分数值发散，留待后续逐例。
