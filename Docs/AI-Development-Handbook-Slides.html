Vertex Shader：把整套光照模型搬进解释器

event399 的 VS 在顶点级算完了环境光 + 漫反射 + 镜面高光 + 距离衰减 + 聚光锥，还带 ColorMaterialMode 分支选色：

// 裁剪空间位置：手工展开的 mul(v0, WorldViewProj)，按列 swizzle + MAD 链
r0 = WorldViewProj._m01_m11_m21_m31 * v0.yyyy;
r0 = v0.xxxx * WorldViewProj._m00_m10_m20_m30 + r0;   ...   o0 = ..._m03... + r0;
// 法线归一化 → 变换到世界空间 → 再归一化（dot/rsqrt 自展开）
r0.x = rsqrt(dot(v1.xyz, v1.xyz)); r0.xyz = v1.xyz * r0.xxx; ...
r2.xyz = LightPos.xyz + -r1.xyz;        // 一元负号：不能当成减法（陷阱#2）
r1.xyz = r0.xyz * -r2.xxx + r1.xyz;     // reflect()
r0.y = exp2(Shininess * log2(r0.x));    // pow 用 log2/exp2 展开
r2 = cmp(ColorMaterialMode == int4(1,5,2,3)); r1.xyz = r2.www ? v2.xyz : MatSpecular.xyz;

解释器必须正确支持的一整套语义

结构与寻址
4 个带 packoffset 的 cbuffer（含 uint/标量混排）· 矩阵按列 swizzle _m01_m11_m21_m31 · 任意 swizzle/写掩码 · 带语义 out 参数 → 寄存器槽位映射
运算与 intrinsic
MAD 累加链顺序 · dot/rsqrt/log2/exp2/max · cmp(...) 向量比较 + ?: 三元 · int4 比较与位运算 · 一元负号不能误判成减法（陷阱#2）
PS — 贴图采样 × 顶点色
DiffuseTexture.Sample(...)：绑定解析、线性过滤、UV 寻址；再 o0 = v1 * r0（顶点光照色 × 贴图）。
标尺 case
任一环算偏，战锤的高光/底色/纹理走向就与真机对不上——它把验证压力同时压在 VS 光照与 PS 采样上。

第二部分 · 奠基期（OpenCode + minimax-m2.7）

案例一 · 找不到根因 · #35

  • 现象：整段 HLSL 函数体被当成一条语句执行 → 失败。
  • AI 的误判：认定按 ; 切分的字符循环写错了，反复改循环。
  • 真因（用户点出）：body 没去掉最外层 { } → 第一字符就让 brace_count=1，顶层 ; 永不切分。循环没错，错在数据。
body = body.strip()
if body.startswith('{') and body.endswith('}'):
    body = body[1:-1]        # 去掉最外层 { }，再按 ; 切分

案例二 · 修复不彻底 · #37

  • 现象：dist <= LightRadius 的 <= 被识别成 <，右操作数错成 = LightRadius。
  • AI 的误判（修了没修对）：加了两字符判断，但取 expr[i-1:i+1]——指向 < 时取到 " <"（空格+<），仍退回单字符。
  • 真因 + 手改：应向前看 expr[i:i+2] 命中 <=，并给单字符加护栏。

提交信息如实记成 "MiniMax-M2.7 fixes two char operator issue failed. I have to fix it by my hand."

案例三 · 误删代码 · #42

  • 任务：重构 if/else，执行前先检查有无 else、有则先合并再执行。

用真实日志说清 X == Y

Y = golden 行数（= 顶点数）；X = 每字段每分量都在容差内的行数。任一分量超差 → 整行失败。

✅ 全通过 event371（6 顶点）

Total PASSED rows: 6/6
Comparison PASSED

❌ 未修好 event399（696 顶点）

Error: Row0 Color[0] output=0 golden=0.3647
Total PASSED rows: 0/696
Comparison FAILED

X==Y 才说明"每个顶点、每个分量都和真机一致"；差一个分量即 X<Y，回归立即判红，且必有对应 Error: 行。

6 · 回归测试的建立与应用

10 · 优缺点与选型启示

第二部分 优点
步幅小、易回退、每步可肉眼验证、成本低；为整个项目的可观测性打了地基。
第二部分 缺点
找不到深层根因、修复不彻底、会误删代码；重度依赖人工兜底、无回归网、推进慢。
第三部分 优缺点
优：长上下文持有整条管线、根因导向、闭环交付、能自我纠错。缺：成本高、对提示词约束敏感（缺"不要做什么"会改错地方）。
选型启示：搭骨架/能力有限时，把需求切碎、靠人验证最稳；攻坚/上下文充足时，让 AI 持有全局、用回归 + 记忆闭环交付最高效。前期的可观测性铺垫，正是后期能"沿数据流快速回溯根因"的前提。
