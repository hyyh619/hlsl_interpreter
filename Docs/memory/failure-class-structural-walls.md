The remaining `Dump/` VS failures bottom out on a few STRUCTURAL walls, not simple bugs. Confirmed in steps 154–156; re-triaged in step 168 (44 zips remain, 9 classes — see Sessions/hlsl-interpreter-step168-*.md for the full per-class case list).

**Step 168 corrections (dumps were RE-EXPORTED with real SRV view formats):**
- `buffer_params.csv` now carries a `ViewFormat` column (real DXGI format) — `_decode_view_format` in hlsl_interpreter.py decodes typed buffers by it directly; the byte-distribution inference is now only a fallback for old dumps / unsupported formats (R32_TYPELESS, R10G10B10A2). The old inference was verified correct on every checked Octopath buffer, so past "format misinference" theories were wrong.
- **The "quaternion/bone-basis component permutation" (event2135/2912) was NOT a decode bug — it was a parser bug, FIXED in step 168:** `_parse_method_call` used `rfind(')')` so a trailing swizzle after a method call (`tex.Load(...).y`) was silently dropped, returning the full float4 and scalarizing to `.x` — which looks exactly like an x↔y swap. Hidden for so long because decompiled code almost always uses prefix-identity swizzles (`.xy`/`.xyzw`). Fixed with a `swizzle` syntax-tree node. event2135 (6/6), event2912 (504/504), sekiro2_event15481/16052 (3/3) all now PASS and are promoted to the regression list (129/129).
- Also fixed: floor/ceil/trunc/round/frac crashed on NaN/±inf (`math.floor` raises); now GPU pass-through semantics (frac(±inf)=NaN). witcher3_event16834 no longer exits 1.
- **"Octopath 576/2651/2682 degenerate terrain" — SOLVED in step 172** (see the step 172 entry above): 5 stacked interpreter bugs (asint negate modifier, asint swizzle raw bits, missing rcp, OOB Load, AddressW-clamped explicit LOD). All three PASS 23064/23064 and are promoted.

**Walls (do NOT re-chase as fixable bugs):**
- **Struct-array matrix SELECTOR drop (C/TombRaider) — FIXED in step 160, NOT a wall.** 3Dmigoto drops the matrix-member selector of a struct-array cbuffer (`struct{float4x4 A,B,C;}Arr[N]` → `Arr[i]._m10..` with the member gone; and `struct{float4x4 M[K];}Arr[N]` → flat `Arr[flat]._m10..`). Recovered from the disasm: `recover_struct_array_matrix_selectors()` (in render.py, after cbuffer parse) pairs member-less HLSL accesses with their `cb<slot>[reg+N]` disasm reads 1:1 in program order, member base = N−R, and rewrites `Arr[i].World._m10..`; the flat single-matrix-array case is split at runtime in get_value (element=flat//K, inner=flat%K). TombRaider 0→29/37, zero regression. So the "selector" loss is recoverable, NOT beyond the boundary.
- **Matrix major-order — LARGELY FIXED in step 161 (per-declared-major-order loading), not a wall.** The bug was in `override_cbuffers_from_binary`: it transposed EVERY float4x4 to logical rows (assuming column-major register storage). Correct for a plain/`column_major` matrix (registers are columns; 3Dmigoto emits the `_m00_m10_m20_m30` column-swizzle which then reads register R), but WRONG for a `row_major` matrix (registers already are rows; 3Dmigoto emits the `_m00_m01_m02_m03` row-swizzle which should read register R directly). Fix: `FieldDefinition.is_row_major` (captured in parse_cbuffer), and the float4x4/float4x3 branches store RAW registers for row_major and transpose only column_major. Both `_mRC` and `mul_matrix_vector` then compute `sum_i v[i]*reg_i[j]`, matching the GPU. TombRaider 29→34, zero regression (the column_major Collision/Frame/sekiro/heaven/valley cases still transpose and pass). NOTE: trying to remove the transpose unconditionally collapses the regression to 84/128 — it MUST be conditional on declared major order. Nobu586 (step 158) was likely this same bug; re-check it. Remaining edge: a row_major matrix with only CSV (no bin) still loads the CSV's transposed .row values — relies on the binary override to correct it.
- **Golden CSV "multi-trailing-float3 misalignment" was a MISREAD (corrected step 158):** `load_vs_golden_from_mesh_csv` already handles N consecutive float3/reduced-width outputs correctly via SV-Position-first ordering + per-output dumped-width (`_dumped_count` from header column groups). event3502 (two consecutive 3-col outputs) passes 384/384; a reconciliation guard fired zero misalignment warnings across all 86 Dump cases. There is NO column-remap gap. The earlier "SV_POSITION data under a TEXCOORD column" observation is exactly what the SV-first reorder fixes. The real remaining D failures are NOT column issues: event2135/2912 are a quaternion/bone-basis component permutation (interpreter decode bug — per-output x<->y swap / 3-cycle), event664/3012 are sv_position skinning/precision, event576/2651/2682 are degenerate terrain (zero cbuffers).
- **No golden** (class B, 16 cases): no comparable VS mesh / 0 vertices — unverifiable by design.

**Step 177 (witcher noise-hash RAW-BITS semantics, regression 147/147):** witcher3_21979 132/840→PASS 840/840 (promoted); 22049 132→790/840.
