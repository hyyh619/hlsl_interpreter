The interpreter now runs **stages beyond VS** and compares each against RenderDoc's exact bin+layout mesh-output golden. Context for extending this (DS/HS/tessellation is the planned next step).

**Mesh-output golden (step 165):** `<name>_<stage>_mesh.bin` + `_<stage>_mesh_layout.csv` (Stage/Stride/NumVerts + per-attr SemanticName,Index,ComponentCount,VarType). `HLSLInterpreter.load_mesh_output_golden(bin, layout)` decodes it (keyed by canonical semantic, same keys the results use); `find_stage_mesh_dump(folder, stage)` locates the pair. Exact and unambiguous — prefer over the `_vs_mesh.csv`.

**GS execution (step 166):** `render._run_gs_stage` builds a GS interpreter, runs the GS over VS-output primitives, compares to `_gs_mesh.bin`. Key pieces in hlsl_interpreter.py: `_assemble_primitives(idx_list, topology)` (triangle-strip winding: even tri (i,i+1,i+2), odd (i,i+2,i+1)); `executeGS_with_params` (GS input slot → VS-output key by semantic; SV_VertexID = drawn index, SV_InstanceID = instance); `_execute_gs_main` (sets `v[i][j]` 2D per-primitive-vertex input; runs main); `stream.Append()`/`RestartStrip()` intercepted in `execute_statement` when `self._gs_emit` is set; `get_value` handles nested `v[i][j].swz`. Passthrough GS (Octopath event102/event4014) pass 6/6.

**GS emit classes SOLVED in step 169 — the "cbuffer decodes wrong" theory was FALSE** (birth indices decoded correctly all along; 411/412 IS the real birth range and the failing rows match it exactly). The real causes were 8 independent bugs, all fixed (regression 130/130):
- manhattan "GS" is a **VS+Stream-Output** draw (disasm `vs_4_0`, GS_shader.hlsl byte-identical to VS): `_run_gs_stage` now passes VS results through when the source has no `Stream<>`/`.Append(`, and trims the golden's all-zero SO-capacity tail. event50 PASSes (promoted); 87/124/161/198 fail only their 1-2 birthed rows = `frac(87362*sin(~80000))` big-phase hash → transcendental wall.
- StructuredBuffer struct members were killed by `// Offset:` comments (stride 176→16); stride now also overridden by buffer_params.csv ElementByteSize (`_captured_sb_strides`).
- SB member trailing swizzle was dropped (`t0[i].Position.y` → whole float4 → scalarized to .x — the "broadcast sv_position" signature).
- **Non-indexed draws: SV_VertexID starts at 0** (StartVertexLocation offsets only VB fetch — D3D11, opposite of Vulkan/GL). VertexOffset=1 had shifted every particle.
- **RenderDoc post-GS golden expands trianglestrips to triangle lists** (rows = strips×6 for quad strips; odd triangle = (i, i+2, i+1)). executeGS_with_params collects per-strip and expands when the disasm says `dcl_outputtopology trianglestrip`.
- `f32tof16` must round TOWARD ZERO (`_f32_to_f16_rtz`), not nearest-even.
- Primitive-element SBs (`StructuredBuffer<uint3>`, `[i].xy` swizzles the element), matrix members (`._mRC` column-major c*4+r), and the 3Dmigoto artifact `rX = rX ? 0.000000 : 0` (= `and rX,rX,0xffff`) repaired in preprocess_hlsl.
Residuals: sekiro2_14998 → 28974/29148 (TexCoord4 ~0.13, precision class); sekiro4_20560 count-exact, only TEXCOORD9 wrong = irradiance volume (Texture3D) RGBM alpha reads 1.0 → exp2 blowup (texture-decode class).

**Output-register convention:** RenderDoc inits each 4-component output register to (0,0,0,1); `_execute_void_main`/`_execute_gs_main` init float4 outputs to `[0,0,0,1]` so unwritten lanes match.

**HS/DS + tessellation (step 167):** `tessellator.py` = standalone fixed-function tessellator `tessellate(domain, edge_factors, inside_factors, partitioning)` → (points, prims) for tri/quad/isoline; integer factor 1 = minimal patch (quad→4 corners, tri→3). `executeHS_with_params` runs the HS control-point phase (empty decompiled body → identity passthrough of VS-output control points). `executeDS_with_params`/`_execute_ds_main` run the DS once per domain point per patch, setting `vicp[i][j]` (2D control points, reuses GS's `v[i][j]`), `vDomain`/`SV_DomainLocation`, and `vpc0..7` (patch constants, default 0). `render._run_ds_stage` wires VS→HS→tessellator→DS and compares to `_ds_mesh.bin`. **Key: RenderDoc DS-out golden count = patches × out_cp** (one DS eval per output control point/corner) — take the first `out_cp` minimal-patch corners as per-patch domain locations. All 5 golden cases count-match; framework proven by sekiro4 Color.xyz exact match (`o0.xyz=vicp[0][5]*vicp[0][1]`) + witcher geometry `vDomain*size+corner` (diff==size).

**Step 183 — witcher _ds_mesh golden layout SOLVED (16803/16817 PASS+promoted, 21346 855/1024, regression 152/152):** the "composite multi-corner rows" were a dump-writer artifact: the bin holds the VS OUTPUT STREAM (= HS input CPs) as records at the VS `dcl_output` MASK widths (o1 declared int4 in HLSL but dcl_output o1.x → width 1; int -1 bits = the NaN seen in golden), while the writer copies each row at the DS-OUT layout stride → rows OVERRUN into following records (16803: 16B records read with 60B stride; verified bit-exact sliding-window rows[i][4:]==rows[i+1][:11]). Fix: CP-stream compare truncates to cp_width = sum of VS dcl_output mask widths, gate generalized to in_cp==out_cp (21346 is 4x4), per-semantic lane placement for shared slots (CORNER@.xy SIZE@.z CM@.w), plus the interpreter-wide `_dedupe_duplicate_out_params` preprocess fix: 3Dmigoto names ALL slot-shared extra outputs `pN`, so two semantics collide (both `p0`) and the second body assignment silently destroyed the first's value — renamed p0__dupK by program order (assignment count must equal decl count). 21346's remaining 169 rows are cp[4] (clipmap page index from the per-frame-updated t17 page table): our containment test is provably correct against the dumped rects — t17 is the FOURTH member of the stale-frame-dynamic-resource family (t22, cb12, t1, t17). HS/DS as an interpreter-defect class is now EMPTY.

**Step 181 UPDATE — HS fork/join phases now EXECUTABLE + sekiro4 pair PASSES (regression 150/150):** (1) dxbc_interp.py extended: icb (immediateConstantBuffer) operands, dynamic `o[r0.x+0].x` dests, real `ld/ld_indexable` + `resinfo` via resource hooks. render.`_run_hs_patch_phases` splits HS disasm fork/join phases, runs them per patch/instance through the VM (vicp/vocp/vForkInstanceID inputs, HS cbuffer bins + HS textures wired), collects SV_TessFactor via dcl_output_siv (_SIV_FACTOR_NAMES) and vpc rows, feeding executeDS_with_params(patch_constants=...) — witcher's 464-instruction fork phase runs fine (vpc0.y real now). (2) sekiro4_19857/20244 SOLVED: the step-167 diagnosis was wrong on both counts — SB stride was already fixed by step-169 machinery, and the `_ds_mesh` golden for in_cp=out_cp=1 point-sprite patches is the HS CONTROL-POINT STREAM (every varying golden column = a CP attribute), flat CP floats re-sliced by the DS-out layout with widths from HS disasm dcl_input masks (v[1][9].xyz→3; decompiled HLSL pads all to float4) and multi-semantic slot packing (CORNER.xy+SIZE.z+CM_LEVEL.w share a slot). CP-passthrough compare implemented in _run_ds_stage; both promoted. (3) witcher 16803/16817/21346 STILL BLOCKED on the golden layout: their _ds_mesh rows pack MULTIPLE domain-corner evaluations into ONE row (sv_position=(corner.xy,size,0) at domain(0,0); TexCoord=corner+size at domain(1,0); UInt bit-pattern columns) — needs dedicated forensics of the dump format before comparisons can align.

**HS/DS structural walls (step 167):** witcher3 (16803/16817/21346) DS reads `vpc0.y` = the HS **fork/join (patch-constant) phase** output, which the 3Dmigoto decompiler **drops** (HLSL keeps only the empty control-point phase) → unfixable without that source. sekiro4 (19857/20244) DS: point-sprite SV_POSITION golden is `(0,0,0,w)` (xyz≡0 RenderDoc convention) + tile-based lighting reads `StructuredBuffer<PointLight>` whose struct stride is misdetected as 16B (should infer from struct layout) → `o10`/TEXCOORD10 wrong. sekiro2 tessellation cases have no `_ds_mesh.bin` (no golden).
