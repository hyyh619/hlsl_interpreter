# Step 205 — BlackMyth 骨骼/morph 解码：movc 交换对的并行语义（数据相关的切线错）

Date: 2026-07-08

## 任务

修 step202 定位但未解决的 BlackMyth 骨骼/morph 解码变体：`event9319 / 3256 / 9526 / 9829 /
15293_8040`——法线（TEXCOORD10）对、切线（TEXCOORD11）与世界坐标（TEXCOORD7）错。step202 已
定位到"基向量第 0/1 列与真机不符"，但没找到根因（同段解码代码在 step201 的 `event7117` 上整例通过）。

## 定位：逐语句轨迹 + 基向量幅值

用 `event9319` row 0 的逐语句轨迹，测各基向量幅值（正确应为单位向量）：
- 法线 `r6` → `r18`：`|r18| = 1.0` ✓
- 切线 `r5_basis` → `r17`：`|r17| = 0.57` ✗
- 副切线 `r7_basis` → `r2.xyw`：`0.57` ✗

**基第 2 列（法线）正确、第 0/1 列（切线）幅值只有 ~0.57**。往回查切线重构，问题出在这条
3Dmigoto 把 DXBC `movc` **交换对**渲染成一行的语句：

```hlsl
r0.x = (int)r5.z ? r0.x : r5.w; r2.x = (int)r5.z ? r5.w : r0.x;
```

这是 GPU 上**并行执行的两条 `movc`**——各自读**旧的** `r0.x`。语义（cond=FALSE 时）：
- `r0.x_new = cond ? r0.x_old : r5.w = r5.w`
- `r2.x_new = cond ? r5.w : r0.x_old = r0.x_old`（读**旧** `r0.x`）

但解释器把一行拆成两条**顺序**语句：第 1 条先把 `r0.x` 改成 `r5.w`，第 2 条再读**新** `r0.x`
（=`r5.w`）→ `r2.x` 拿到 `r5.w` 而非 `r0.x_old`。`r2.x` 随后喂 `r5.xyz = r8.xzw * r2.xxx`，
切线基幅值就错了（0.57 而非 1.0）。

### 为什么 `event7117` 能过、`event9319` 不能（同段代码）
条件 `(int)r5.z` 是**逐顶点**的符号位：
- `event7117` row 0：`r5.z = 32768`（TRUE）→ 第 1 条 `r0.x = cond ? r0.x : r5.w` **不改** `r0.x`
  → 第 2 条读到的仍是旧值 → **无冲突**，恰好正确。
- `event9319` row 0：`r5.z = 0`（FALSE）→ 第 1 条**改**了 `r0.x` → 第 2 条读到新值 → **触发冲突**。

所以这是个**数据相关**的顺序冲突：只有当条件选中"会改写 `r0.x`"的分支时才暴露，step201/202
的样本恰好没踩到（或该支被 mask 成 0）。

## 修复（`hlsl_interpreter.py` `GenerateStmts` → 新增 `_rewrite_movc_swaps`）

对 `GenerateStmts` 切出的语句序列做一次后处理：识别**相邻的 movc 交换对**
`d1 = c ? A : B; d2 = c ? B : A;`（同条件、操作数互换、且 `d1` 被第 2 条读到构成冲突、
而第 1 条不读 `d2`），把两条**换序**（先执行"读 `d1` 的那条"），使两条都读到旧 `d1`——
等价于 GPU 的并行语义，且无需引入临时变量。

判定极严格（同 cond + 操作数恰好互换 + 真冲突 + 反向无冲突），单元测试确认只对真正的交换对
生效，普通/不同条件/无冲突的相邻 select 一律不动。

## 验证

- **本家族 6 个全部通过**：`event9319 1050/1050`、`event3256 1065/1065`、`event9526 705/705`、
  `event9829 1050/1050`、`event15293_8040 1050/1050`，外加 step202 的 `event19470_8040 6/6`。
- `event7117`（step201）仍 **6/6**（换序对 TRUE 分支是恒等操作，不受影响）。
- 单元测试：交换对→换序；不同条件 / 未互换 / 无冲突 / 普通赋值→原样不动。

## 结论
根因是 **movc 交换对的并行 vs 顺序语义**——一个**数据相关**的通用控制流 bug，靠"基向量幅值 +
逐语句轨迹 + 跨样本对比（为何 7117 过而 9319 不过）"定位。修复是识别交换对并换序，通用、
不针对 BlackMyth。至此 step201/202/205 三步把 BlackMyth 压缩/骨骼解码的 cast-三元、`;`-else、
B8G8R8A8、movc 交换对四类根因全部修掉。
